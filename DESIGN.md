# ChaosLLM: Design Document v0.1

Chaos engineering for LLM-powered applications. Inject realistic failures into the
path between an app and its LLM / vector DB providers, measure how the app degrades,
and produce a resilience report.

Author: Rhugved Kale
Status: v0.1 scope, locked. Anything not listed here is v2.

---

## 1. Problem

Production LLM apps fail in ways traditional software does not: provider rate limits,
latency spikes measured in seconds, silently degraded model quality, truncated or
malformed outputs, context window overflows. Teams ship RAG apps and agents with no
idea how they behave when the vector DB adds 300ms or the LLM API starts returning
429s. Chaos engineering solved this for microservices (Chaos Mesh, Gremlin, Chaos
Monkey). Nothing equivalent exists for the LLM layer.

ChaosLLM is a fault-injection proxy plus an experiment runner. Point your app at the
proxy instead of the provider, define experiments in YAML, run them, get a report.

## 2. Non-goals for v0.1

- No Kubernetes operator. Single process, Docker Compose for the demo stack.
- No automatic fix suggestions. The report states facts; remediation is a docs page.
- No hallucination "detection" as a research problem. Output quality is checked via
  simple, deterministic assertions (see 6.4).
- No auth/multi-tenancy. This is a developer tool, not SaaS.
- No streaming fault injection in v0.1 (pass streaming through untouched). v2.

## 3. Architecture

```
                        +---------------------------+
                        |     chaosllm proxy        |
  +----------+  HTTP    |  +---------------------+  |   HTTP   +----------------+
  | Your app | -------> |  | fault middlewares   |--|--------> | LLM provider   |
  | (or demo | <------- |  | latency|errors|...  |  | <------- | (OpenAI/       |
  |  RAG app)|          |  +---------------------+  |          |  Anthropic/...)|
  +----------+          |     |  metrics tap        |          +----------------+
       ^                +-----|---------------------+
       |                      v
       |                +-----------+     +------------------+
  load generated by     | SQLite +  | <-- | experiment runner |  <- experiment.yaml
  the runner            | JSONL log |     | (async load gen)  |
                        +-----------+     +------------------+
                              |
                              v
                     +------------------+        +-----------------+
                     | report generator | -----> | dashboard (SSE) |
                     | (md + json)      |        | React, live     |
                     +------------------+        +-----------------+
```

Three deployables, one repo:

1. `chaosllm` Python package: proxy + runner + report CLI.
2. `demo/` victim app: a small RAG service used for the live demo.
3. `dashboard/` React app: live experiment view, reads the control API via SSE.

## 4. Components

### 4.1 Fault-injection proxy

- ASGI app (FastAPI + httpx.AsyncClient) acting as a reverse proxy.
- Provider-aware routes: `/openai/*` forwards to `https://api.openai.com/*`,
  `/anthropic/*` to `https://api.anthropic.com/*`, plus a generic
  `/passthrough/{base_url_id}/*` configured in `proxy.yaml` for vector DBs or any
  HTTP upstream (e.g. a hosted Chroma/pgvector service, or the demo app's own
  retrieval endpoint).
- The app under test needs ONE change: set its provider base_url to the proxy.
  Both OpenAI and Anthropic SDKs support base_url override. This is the core UX win.
- Faults are middlewares in a pipeline. Each request passes through the active fault
  set; each fault decides independently (probability per fault) whether to fire.
- Fault state is set at runtime by the runner via a control API
  (`POST /control/faults`), so one long-lived proxy serves many experiments.
- Every request/response is logged to the metrics tap: timestamp, route, upstream,
  fault(s) fired, latency (proxy-added and upstream), status, token counts when
  parseable from the response body, request id.

### 4.2 Fault types (v0.1, complete list)

| id                | What it does                                                      | Key params                          |
|-------------------|-------------------------------------------------------------------|-------------------------------------|
| `latency`         | Sleep before forwarding and/or before returning response          | `delay_ms`, `jitter_ms`, `p`        |
| `error`           | Return a synthetic provider error without calling upstream        | `status` (429/500/503), `body`, `p` |
| `timeout`         | Accept request, never respond until client gives up (blackhole)   | `hold_ms` cap, `p`                  |
| `truncate`        | Forward normally, cut response body at N% of bytes                | `keep_fraction`, `p`                |
| `malformed_json`  | Forward normally, corrupt the JSON (drop closing brace, mangle key)| `mode`, `p`                        |
| `context_overflow`| Return the provider-shaped "context length exceeded" error        | `p`                                 |
| `model_downgrade` | Rewrite the `model` field in the request to a weaker model        | `to_model`, `p`                     |
| `rate_limit_burst`| Stateful: after N requests in window, return 429 with retry-after | `limit`, `window_s`                 |

Every synthetic error mimics the real provider's error shape (status, JSON body,
headers like `retry-after`) so SDK retry logic reacts exactly as it would in prod.
This fidelity is a headline feature; get the shapes from real provider docs and
pin them in `chaosllm/faults/provider_shapes.py` with source links in comments.

### 4.3 Experiment spec (YAML)

```yaml
# experiments/vector-db-slow.yaml
name: vector-db-latency-spike
description: RAG behavior when retrieval latency exceeds 200ms
target:
  base_url: http://localhost:8100        # app under test
  endpoint: POST /ask
  payload_file: payloads/questions.jsonl # one JSON body per line, runner cycles them
load:
  concurrency: 8
  duration_s: 120
  warmup_s: 15                           # baseline window, no faults
faults:
  - id: latency
    route: /passthrough/vectordb/*
    delay_ms: 300
    jitter_ms: 100
    p: 1.0
assertions:
  - type: success_rate
    min: 0.95
  - type: latency_p95_ms
    max: 4000
  - type: response_contains          # graceful-degradation marker
    field: answer
    forbid_empty: true
  - type: json_field_present         # RAG answers must still cite sources
    field: citations
    min_items: 1
```

The spec format deliberately mirrors Chaos Mesh / Gremlin conventions (named
experiment, blast radius via route matching, steady-state assertions). Reviewers
who know chaos engineering should feel at home.

### 4.4 Experiment runner

- Pure asyncio load generator (no Locust: fewer deps, fully explainable, and the
  interview question "why not Locust" has a good answer: we need per-request fault
  correlation and a warmup/fault/recovery phase machine, which is simpler to own).
- Phases: `warmup` (baseline, faults off) -> `chaos` (faults on) -> `recovery`
  (faults off, same duration as warmup). Every metric is reported per phase; the
  baseline-vs-chaos delta IS the resilience finding.
- Runner drives load at the target app, not the proxy. The proxy only sees the
  provider-bound traffic that the target app generates.
- Results: JSONL event log per run + summary row in SQLite
  (`runs`, `requests`, `assertions` tables). Run id = `{name}-{timestamp}`.

### 4.5 Resilience report

- `chaosllm report <run_id>` renders Markdown (and `--json`).
- Contents: experiment summary, per-phase table (success rate, p50/p95/p99,
  error taxonomy), assertion pass/fail, fault-fire counts, and a
  "degradation profile" line chart (latency over time with fault window shaded,
  rendered as SVG for embeddability in READMEs).
- The money artifact: sentences like
  "Success rate dropped from 99.4% (baseline) to 61.2% under 300ms vector DB
  latency; 84% of failures were client timeouts, 16% empty answers."

### 4.6 Demo victim app (`demo/`)

- Minimal RAG service: FastAPI, in-process vector store (chromadb), a corpus of
  ~50 short docs (use public-domain text), `POST /ask` -> retrieve -> LLM ->
  `{answer, citations[]}`.
- Two builds behind a flag: `RESILIENT=false` (naive: no timeouts, no retries, no
  fallbacks) and `RESILIENT=true` (httpx timeouts, tenacity retries with jitter,
  circuit breaker on the LLM client, cached-embedding fallback, "degraded answer"
  path that returns retrieval-only extractive answers when the LLM is down).
- The live demo story: run the same experiment against both flags, show the two
  reports side by side. This demonstrates the tool AND teaches resilience patterns.

### 4.7 Dashboard (`dashboard/`)

- React + Vite, single page. Connects to the control API via SSE
  (`GET /control/runs/{id}/events`).
- Shows: current phase, live success-rate and p95 sparkline, fault-fire ticker,
  assertion status. Nothing else in v0.1.
- Styling: clean, dark, one accent color. This is a screenshot machine for the
  README and the blog post; design accordingly.

## 5. Tech stack (locked)

- Python 3.12, FastAPI, httpx, pydantic v2, PyYAML, tenacity (demo app only),
  chromadb (demo app only), SQLite via sqlite3 stdlib (no ORM: explainability).
- React 18 + Vite + plain CSS for the dashboard.
- pytest + pytest-asyncio + respx (mock upstreams); ruff + mypy strict on
  `chaosllm/` package.
- Docker Compose: `proxy`, `demo-app`, `dashboard`. GitHub Actions: lint, type,
  test on PR; build and push images on tag.
- Deploy: Railway for proxy + demo app + control API; dashboard static on Vercel.
  Hosted demo uses a low-cost model (e.g. claude-haiku / gpt-4o-mini class) with a
  hard daily budget cap enforced in the proxy (spend tracker, returns 402 past cap).

## 7. Definition of done for v0.1

1. `pip install -e . && chaosllm demo up` starts proxy + demo app locally.
2. `chaosllm run experiments/vector-db-slow.yaml` completes and prints a report.
3. Same experiment vs `RESILIENT=true` demo shows a materially better report.
4. Live deployment: dashboard URL anyone can open, watch a scheduled hourly
   experiment run against the hosted demo app.
5. README: 90-second pitch, architecture diagram, demo GIF, quickstart that works
   on a clean machine, honest "limitations" section.
6. Test coverage: every fault type has a unit test proving its injected shape
   matches the pinned provider shape; runner has an end-to-end test against a
   respx-mocked upstream.
7. Owner (Rhugved) can explain every module without notes. Enforced via PLAN.md
   checkpoints.
