# chaosllm

Chaos engineering for LLM-powered applications.

Production LLM apps fail in ways ordinary software doesn't: provider rate limits,
multi-second latency spikes, silently truncated or malformed output, context window
overflows. Chaos Mesh and Gremlin solved this for microservices. Nothing equivalent
existed for the LLM layer, so this is that: point your app's `base_url` at the
chaosllm proxy instead of the provider, define an experiment in YAML (which routes,
which faults, how long), run it, and get a report that says exactly how your app
degraded and by how much.

The proxy also enforces a hard daily spend cap and mimics real provider error shapes
(status codes, JSON bodies, `retry-after` headers) closely enough that your SDK's own
retry logic reacts to injected faults the same way it would to the real thing.

## Architecture

```
                        +---------------------------+
                        |     chaosllm proxy        |
  +----------+  HTTP    |  +---------------------+  |   HTTP   +----------------+
  | Your app | -------> |  | fault middlewares   |--|--------> | LLM provider   |
  | (or demo | <------- |  | latency|errors|...  |  | <------- | (OpenAI/       |
  |  RAG app)|          |  +---------------------+  |          |  Anthropic/...)|
  +----------+          |     |  metrics tap        |          +----------------+
       ^                +-----|---------------------+
       |                      v
       |                +-----------+     +------------------+
  load generated by     | SQLite +  | <-- | experiment runner |  <- experiment.yaml
  the runner            | JSONL log |     | (async load gen)  |
                        +-----------+     +------------------+
                              |
                              v
                     +------------------+        +-----------------+
                     | report generator | -----> | dashboard (SSE) |
                     | (md + json)      |        | React, live     |
                     +------------------+        +-----------------+
```

Three deployables, one repo:

- `chaosllm/`: the Python package. Fault-injection proxy, experiment runner, report
  generator, CLI (`chaosllm proxy | run | report | demo`).
- `demo/`: a small RAG service (FastAPI + chromadb) used as the app under test for
  the live demo. Has a naive and a resilient build behind a flag; see
  [docs/resilience-patterns.md](docs/resilience-patterns.md).
- `dashboard/`: a React app that watches a run live over SSE.

## Quickstart

Verified on a clean Python 3.12 venv with no prior chaosllm install.

```bash
git clone https://github.com/Rhugved-Kale/chaosllm.git
cd chaosllm
python3 -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"

chaosllm proxy   # starts the fault-injection proxy on :8000, using proxy.yaml
```

In a second terminal, point an OpenAI or Anthropic SDK at the proxy instead of the
real provider:

```python
from openai import OpenAI

client = OpenAI(base_url="http://127.0.0.1:8000/openai/v1", api_key="sk-...")
```

Every request is logged to `metrics.jsonl`, no faults active yet. To see the full
story (fault injection, phased load, a resilience report), run the demo stack
instead:

```bash
cp .env.example .env   # fill in OPENAI_API_KEY or ANTHROPIC_API_KEY + LLM_PROVIDER
chaosllm demo up       # docker compose: proxy + demo-app + dashboard
chaosllm run experiments/llm-latency-spike.yaml
```

This drives load at the demo app's `/ask` endpoint through three phases (warmup,
chaos, recovery), injects 2-second latency spikes and rate-limit bursts on the
Anthropic route during the chaos phase, and prints a Markdown report: success rate
and p50/p95/p99 latency per phase, assertion pass/fail, fault-fire counts. Open
`http://localhost:5173` before the run to watch it live.

Run the same experiment again with `RESILIENT=true` in `.env` and compare reports;
that delta is the whole pitch.

```bash
chaosllm demo down   # tear down when done
```

## Limitations (v0.1)

- No streaming fault injection: a `stream: true` request is passed through
  untouched. Faulting a live token stream is a real feature, just not this version's.
- No Kubernetes operator or multi-node load generation. Single-process asyncio load
  generator; enough to demonstrate the resilience delta, not a load-testing tool.
- No hallucination or output-quality scoring. Assertions are deterministic (response
  non-empty, a field present, latency under a threshold), not a judgment on whether
  the answer was actually good.
- No auth or multi-tenancy. This is a developer tool you run yourself, not a hosted
  service with accounts.
- The provider error shapes in `chaosllm/faults/provider_shapes.py` are pinned from
  current OpenAI and Anthropic docs at the time they were written. Providers change
  error formats without much notice; treat these as a snapshot, not a guarantee.
- The hosted demo runs a low-cost model under a hard daily budget cap
  (`BUDGET_DAILY_USD`) and will return 402 once it's spent for the day. That's by
  design, not a bug.
